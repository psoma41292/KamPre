KamPare – High-Level Technical Summary

Prepared for stakeholder review

Overview

KamPare is a grocery price comparison web application that helps users search for common grocery items and compare prices across multiple retail platforms. Its goal is to make price checking faster, highlight the best available deal, and support informed buying decisions. The application also supports user login, saved favorites, and linked platform accounts for a more personalised experience.

Architecture Summary

The system uses a simple client-server architecture. The frontend is a single-page React application that handles search, result views, authentication screens, and user interactions. The backend is a Node.js and Express API that exposes endpoints for comparison, categories, favorites, grocery catalog data, and authentication. Business logic is handled in a service layer, which calculates best deal and best value metrics, groups products by quantity, and caches responses for faster repeat searches. Data currently comes from a mock product dataset, with optional live price lookup through SerpAPI. Favorites and some user-related features are currently stored in memory, while MongoDB support exists as an optional persistence path.

Key Technologies

  • React and Vite for the frontend user interface
  • Tailwind CSS for styling
  • Axios for frontend-backend API communication
  • Node.js and Express for backend services
  • NodeCache for in-memory caching
  • JSON Web Tokens for authentication
  • MongoDB and Mongoose as optional database components
  • Recharts for price visualisation

Core Components

  • Search and suggestion module: captures user queries and returns matching items
  • Comparison engine: retrieves product data, normalises quantities, and ranks deals
  • Results interface: shows cards, tables, charts, and best-deal summaries
  • Authentication module: manages registration, login, token-based sessions, and account linking
  • Favorites module: saves and removes frequently searched items
  • Catalog and category services: provide browseable grocery and category data

Data Flow

A user enters a grocery item in the frontend search bar. The frontend sends the request to backend API endpoints using Axios. The backend processes the request through Express routes and the price service, which fetches live or mock data, enriches it with calculated values such as price per unit, determines the best options, and caches the result. The response is then returned to the frontend, where it is displayed as banners, cards, tables, and charts. For logged-in users, authentication tokens are included in API requests so account and favorites features can be used.
